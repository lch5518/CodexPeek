# Task 4 네이티브 Windows 애플리케이션 보고서

## 1. 문제 분석

기존 Windows 앱은 Explorer 재시작 뒤 작업 표시줄 자식 창이 소멸할 수 있고, 모달 트레이 메뉴 호출 중 Rust 가변 참조가 재진입될 수 있으며, UI 스레드에서 설정 저장이나 폴러 재시작이 대기할 수 있었다. 또한 전역 좌표 저장, 프로세스 로캘 기반 언어 선택, 단순 창 존재 기반 작업 표시줄 진단은 다중 모니터와 실제 사용자 환경을 정확히 반영하지 못했다.

## 2. 가정과 위험

- Explorer 내부 창 계층은 Windows 빌드마다 달라질 수 있으므로 알려진 알림 영역을 찾되, 실제 수평 공간 배치 계산까지 통과한 작업 표시줄만 호환으로 판단한다.
- Windows 10의 낮은 작업 표시줄 높이에서는 상태 문구를 생략하는 2행 축약 렌더러를 사용한다.
- 수동 Windows 10/11, 혼합 DPI, Explorer 재시작 검증은 별도 QA가 필요하며 이 보고서에는 자동 검증 결과만 기록한다.
- 업데이트 검사는 정확한 GitHub 태그 URL만 열고 동시에 둘 이상의 검사를 실행하지 않는다.

## 3. 해결 설계

- 메시지 전용 창 대신 숨겨진 최상위 도구 창을 트레이 메뉴와 `TaskbarCreated` 메시지의 안정적인 소유자로 사용한다.
- 창, 타이머, 트레이 소유권을 순수 `NativeLifecycle` 상태로 모델링하고 모든 생성 이후 오류 경로에서 한 정리 함수가 역순으로 해제한다.
- `TrackPopupMenu` 같은 재진입 가능 Win32 호출 전에는 상태를 복사하고 Rust 참조를 해제한 뒤, 반환 후 원시 포인터로 상태를 다시 확인한다.
- 단일 인스턴스 잠금은 설정, 폴러, 업데이트 검사보다 먼저 획득한다. 진단 모드만 의도적으로 우회한다.
- 폴러는 명령 채널로 갱신 간격과 자동 인증 정책을 변경하고, 설정은 단일 비동기 작성기에서 제출 순서대로 저장한다.
- 작업 표시줄 연결과 해제를 API 결과와 실제 부모 창으로 검증한다. 해제를 검증하지 못하면 창을 폐기하고 다시 만든다.
- 부동 좌표는 대상 모니터 작업 영역 원점 기준 논리 좌표로 저장하고 대상 모니터 DPI로 복원한다.
- Windows UI 언어 ID를 우선 사용하고, 진단은 CLI, 앱 서버, 로그인, 응답 형식을 분리하여 콘솔과 현지화된 대화 상자에 표시한다.

## 4. 구현

- 숨겨진 최상위 소유자 창, Explorer 복구, `WM_NCDESTROY` 핸들 무효화, 중앙 정리를 구현했다.
- 트레이 메뉴는 정적 호출로 바꾸고 실제 소유자 활성화 및 메뉴 종료용 `WM_NULL` 게시를 추가했다.
- 검증된 작업 표시줄 연결/해제, `SWP_FRAMECHANGED`, 즉시 부동 배치 및 실패 시 재생성을 구현했다.
- 음수 원점과 혼합 DPI를 포함하는 모니터 상대 좌표 저장/복원을 구현했다.
- `AsyncSettingsWriter`와 런타임 폴러 재구성 명령을 추가해 UI 스레드 대기를 제거했다.
- Windows UI 언어 선택, 안전한 CLI 위치 진단, 현지화 진단 요약, 실제 작업 표시줄 기하 호환성 검사를 추가했다.
- 축약 렌더러의 여백, 간격, 막대 두께, 텍스트 예약 폭을 DPI에 맞춰 스케일링했다.
- 작업 표시줄 연결을 스타일 변경, 부모 변경, 위치 변경의 검증된 트랜잭션으로 만들고 모든 실패 단계에서 원래 부모와 스타일을 복구한다. 스타일 설정의 0 반환값은 신뢰하지 않고 실제 스타일을 다시 읽어 확인한다.
- 48 논리 픽셀 높이를 모두 수용하지 못하는 작업 표시줄과 음수 오프셋을 거부하여 부동 창으로 안전하게 대체한다.
- `PollingService::drop`은 중지 신호만 전송하고 작업자 핸들을 분리한다. 진행 중 공급자 호출은 자체 제한 시간과 자식 프로세스 가드를 유지하며, 완료 후 대기 중인 중지 신호를 처리해 작업자 리소스를 회수한다. 명시적 `stop`은 테스트와 정상 동기 종료를 위해 계속 join한다.
- 작업 표시줄 연결 순서와 롤백을 작은 `TaskbarAttachmentBackend` 트랜잭션으로 분리했다. 실제 Win32 어댑터와 호출 기록용 테스트 backend가 같은 함수를 사용하므로 스타일·부모·위치 순서 또는 단계별 롤백이 바뀌면 회귀 테스트가 실패한다.

## 5. 사용 및 테스트 예

```powershell
cargo build
target\debug\codex-usage-monitor.exe
target\debug\codex-usage-monitor.exe --startup
target\debug\codex-usage-monitor.exe --diagnose
```

자동 검증 결과:

- `cargo test --all-targets`: 100개 통과, 실패 0
- `cargo clippy --all-targets --all-features -- -D warnings`: 통과
- `cargo fmt --all -- --check`: 통과
- `cargo build`: 통과
- `git diff --check`: 통과

추가된 회귀 테스트는 복구/정리 순서, 해제 실패 재생성, 시작 순서, UI 언어, 정확한 릴리스 URL, 음수 원점·혼합 DPI, 순차 비동기 설정 저장, 비차단 폴러 재구성, 낮은 작업 표시줄 거부, 음수 오프셋 거부, 진행 중 공급자 호출에서의 즉시 Drop, 연결 성공 순서, 다섯 실패 단계의 상태·프레임 복구, 롤백 자체 실패 보고를 다룬다.

## 6. 가능한 개선

- Windows 10/11과 100/125/150/200% DPI 조합에서 실제 작업 표시줄 연결, Explorer 재시작, 다중 모니터 이동을 수동 검증한다.
- 향후 Windows 작업 표시줄 계층 변경이 확인되면 창 클래스 이름을 늘리기보다 검증된 탐색 규칙을 별도 어댑터로 추가한다.
- 백그라운드 설정 저장 실패를 사용자에게 지속적으로 노출할 필요가 생기면 마지막 저장 오류 상태를 진단 요약에 연결한다.
